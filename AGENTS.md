# AGENTS.md

## 目的

本文件用于规范参与本项目的 AI 代理与开发者的协作方式，目标是：

- 让改动边界清晰、可追踪
- 让文档与代码保持同步
- 让交接成本可控
- 降低在 Windows、本地代理、微信自动化场景下的误操作风险

本文件面向：

- AI 代码代理
- 人类开发者
- 后续接手项目的维护者

## 项目定位

这是一个运行在 Windows 上的本地签到辅助项目，核心链路包括：

- `web.py` 提供本地 Web 服务
- `uiautomation` 模式支持旧版微信
- `cv + mitmproxy` 模式支持新版微信
- 项目依赖本地微信界面、系统代理、证书状态和模板匹配结果

因此本项目的开发原则是：

- 先保证可观察
- 再保证可恢复
- 最后再做更强的自动化和封装

## 文档分工

修改代码时，必须明确判断是否要同步更新以下文档：

- [README.md](/D:/wei-class/README.md)
  用户使用文档。凡是影响部署、配置、启动方式、环境变量、模板目录、诊断方式的改动，都必须更新。
- [PROJECT_PLAN.md](/D:/wei-class/PROJECT_PLAN.md)
  路线图文档。凡是影响阶段目标、优先级、里程碑顺序的改动，都应更新。
- [IMPLEMENTATION_BACKLOG.md](/D:/wei-class/IMPLEMENTATION_BACKLOG.md)
  执行清单。凡是完成了 backlog 任务、拆分了任务、阻塞了任务、调整了依赖关系，都应更新状态。
- `AGENTS.md`
  协作规范。凡是改变了开发流程、提交方式、交接要求、验证基线，都应更新。

默认规则：

- 改用户可见行为：更新 `README.md`
- 改路线或优先级：更新 `PROJECT_PLAN.md`
- 改任务状态或执行顺序：更新 `IMPLEMENTATION_BACKLOG.md`
- 改协作规则：更新 `AGENTS.md`

## 开发与修改原则

### 1. 以小步、可回滚的改动为主

- 每次只解决一个明确问题或一组紧密相关问题
- 不把不相关的重构混进功能修复
- 尽量保持提交边界清晰

### 2. 不擅自处理无关改动

- 发现工作区有无关文件变化时，不主动回退
- 不删除用户文件，除非用户明确要求
- 不覆盖本机模板、日志、配置，除非本次任务明确需要

### 3. 优先保守处理高风险能力

以下内容属于高风险区域，改动时需要格外谨慎：

- Windows 系统代理切换
- 证书导入
- 微信窗口自动化
- 模板自动点击
- `mitmproxy` 启动与链式代理
- 安装包与打包流程

### 4. 先做可观察性，再做自动化

如果某个新功能需要自动恢复、自动修复或自动初始化，优先保证：

- 状态可见
- 错误可见
- 失败后能给出下一步建议

不要先做“黑盒自动化”，再补诊断能力。

## Git 与提交规则

### 基本原则

- 代理默认不自动 commit
- 只有用户明确确认后，才执行 `git add` / `git commit`
- 每次准备提交前，必须先向用户汇总 diff 摘要

### 标准流程

1. 代理完成一项任务
2. 代理运行必要验证
3. 代理汇报：
   - 改了什么
   - 验证结果
   - 风险点
   - 建议提交标题
4. 用户决定：
   - 现在提交
   - 继续累积到下一批
   - 先看 diff

### 提交边界

优先按下面方式拆分 commit：

- 一类功能一个提交
- 一次结构性重构一个提交
- 文档更新可与对应功能一起提交，也可单独提交
- 能独立回滚的内容尽量单独提交

## 验证要求

### 通用要求

每次代码改动后，代理至少应完成以下其中一类验证：

- 运行脚本或接口验证
- 启动服务并检查健康接口
- 针对变更点的手工流程验证
- 如果无法完整验证，必须明确说明原因

### Windows / 微信相关改动

如果改动涉及微信自动化、模板匹配、代理切换、证书、`mitmproxy`，优先验证：

- 服务是否能启动
- `/api/openid_status` 是否可访问
- 当前 `collector_method` 是否正确
- 失败时是否能给出可理解错误
- 是否影响缓存 `openid` 回退逻辑

### 文档改动

如果改动涉及部署方式、命令、环境变量、目录结构、脚本行为：

- 必须同步更新 `README.md`
- 必须检查命令示例是否仍然可执行

## 配置与本地产物规则

### 应进入仓库的内容

- 默认模板 `cv_templates`
- 通用脚本
- 用户文档
- 规划文档
- 默认配置样例

### 不应进入仓库的内容

- `logs/` 下的运行时文件
- 本机采集截图
- 本机调试日志
- 本机生成的 `latest_openid.json`
- 本机覆盖模板 `cv_templates_local`
- 虚拟环境目录

### 配置变更规则

如果新增了任何配置项，必须同时处理：

1. 默认值
2. 配置读取逻辑
3. `README.md` 中的配置说明
4. 必要时更新 `local_config.example.json`

## 文档与交接要求

每次任务完成后，代理应给出简洁交接说明，至少包含：

- 本次完成的目标
- 主要改动文件
- 验证结果
- 残留风险
- 下一步建议

如果任务对应某个 backlog 项，还应说明：

- backlog 编号
- 当前状态应更新为：
  - `doing`
  - `review`
  - `done`
  - `blocked`

## backlog 执行规则

本项目的执行顺序以 [IMPLEMENTATION_BACKLOG.md](/D:/wei-class/IMPLEMENTATION_BACKLOG.md) 为准。

执行时默认规则：

- 新任务默认从 backlog 顺序继续推进
- 若需跳过顺序，必须说明理由
- 若发现计划需要调整，应先更新 backlog，再继续开发

## UI 与产品规则

对用户可见的页面，优先满足：

- 让用户知道当前状态
- 让用户知道下一步要做什么
- 让用户知道失败原因

不要只暴露原始技术异常。

## 禁止事项

除非用户明确要求，否则代理不得：

- 擅自 commit
- 擅自删除用户文件
- 擅自回退不属于本次任务的改动
- 擅自重置 Git 历史
- 擅自把本机日志、截图、缓存结果纳入仓库
