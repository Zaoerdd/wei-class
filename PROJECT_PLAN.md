# 项目计划

## 目标

把当前项目从“开发者可部署、需要手动排障”的状态，推进到“普通 Windows 用户可自助安装、可自助诊断、可稳定运行”的状态。

核心结果：

- 降低首次部署难度
- 降低新版微信 `cv` 模式的人工干预
- 提高失败时的可诊断性和可恢复性
- 为后续打包分发建立稳定基础

## 排序原则

本计划按下面的顺序推进：

1. 先补可观测性
   没有状态展示、健康检查和诊断能力时，后续做再多自动化，出问题都很难定位。
2. 再补配置和模板基础设施
   先把“配置从哪里来、模板从哪里来、优先级如何覆盖”设计清楚，后面的自动采集和向导才不会变成临时拼接。
3. 再补采集稳定性
   先把新版微信采集链路做稳，再做一键初始化和首次向导，避免把不稳定流程包装得更复杂。
4. 最后做强封装分发
   免 Python 打包版应建立在前面流程已经稳定、配置已经收敛的前提上，否则维护成本会迅速升高。

## 总体路线图

### Phase 0：可观测性与支持能力

目标：先让用户和开发者都看得见当前状态，知道为什么失败，知道下一步该做什么。

包含内容：

- 更清晰的运行状态 UI
- 环境体检页
- 一键导出诊断包
- README 分层文档

建议原因：

- 这是所有后续工作的基础
- 能立刻降低支持成本
- 能为后续自动修复和首次向导提供统一状态来源

交付物：

- 首页展示当前 `openid` 来源、上次刷新时间、下次刷新时间、最后错误、当前采集方式、mitm 状态
- 新增部署检查页，逐项显示 Python、微信、代理、证书、mitm、模板、缓存状态
- 新增诊断包导出，输出脱敏日志、健康检查结果、配置快照、模板状态
- 新增 `README.quickstart.md` 和 `README.troubleshooting.md`，保留主 [README.md](/D:/wei-class/README.md)

验收标准：

- 新用户不看终端，也能在页面判断“当前是否可用”
- 用户提交诊断包后，开发者能在一次往返内定位大部分部署问题

### Phase 1：配置与模板基础设施

目标：把“本机差异”从代码里剥离出来，沉淀成明确的配置层和模板层。

包含内容：

- 自动生成本地配置文件
- 模板配置产品化

建议原因：

- 这是自动采集本机模板和首次启动向导的前置条件
- 可以避免后续继续把机器差异写死在代码和 README 里

交付物：

- 首次运行自动生成 `local_config.json`
- 配置文件里明确区分：
  - 基础运行配置
  - PushPlus 配置
  - `cv` 采集配置
  - 模板覆盖配置
- 明确模板优先级：
  - 仓库默认模板 `cv_templates`
  - 本机覆盖模板 `cv_templates_local`
- 模板状态检查接口：
  - 缺哪些模板
  - 哪些模板正在被覆盖
  - 上次模板更新时间

验收标准：

- 用户不需要手动拼 JSON 结构
- 同一份代码可以在不同机器上通过本地配置覆盖运行

### Phase 2：采集稳定性与自恢复

目标：把新版微信的 `cv` 采集做成“失败可恢复、状态可回退、用户少干预”的流程。

包含内容：

- 首次运行时自动截取并生成本机模板
- 错误自动修复

建议原因：

- 这是当前部署难度最高、人工参与最多的部分
- 只有这里稳定后，前面的健康检查和后面的向导才真正有价值

交付物：

- 首次模板采集流程：
  - 自动识别当前微信主窗口
  - 引导用户确认 `微助教服务号` 已可见
  - 自动截取并写入 `cv_templates_local`
- 自恢复能力：
  - 会话已选中时不重复点击
  - 检测到残留浏览器窗口时先关闭再继续
  - 已存在可用缓存 `openid` 时优先保障服务可启动
  - 采集失败时输出明确下一步建议
- 模板失配处理：
  - 自动提示重采哪一张模板
  - 能只重采单个模板，而不是全部重做

验收标准：

- 在新机器上，`cv` 模式首次成功率明显提升
- 用户遇到模板问题时，不需要手工裁图到仓库目录

### Phase 3：首次安装与首次启动体验

目标：把“命令行部署”变成“有顺序、有反馈、有补救路径”的产品化安装流程。

包含内容：

- 一键初始化脚本
- 首次启动向导

建议原因：

- 这一步依赖前面的健康检查、配置基础和采集稳定性
- 前面的基础打牢后，这里可以非常顺滑

交付物：

- 一键初始化脚本，完成：
  - 选择 Python 版本
  - 创建 `.venv`
  - 创建 `.venv-mitm`
  - 安装依赖
  - 生成并导入证书
  - 启动 `mitmproxy`
  - 启动 Web 服务
  - 打开健康检查页
- 首次启动向导，完成：
  - 选择微信模式
  - 填写或跳过 PushPlus
  - 运行环境体检
  - 必要时采集本机模板
  - 执行一次 `openid` 测试采集

验收标准：

- 新用户首次部署可在 10 分钟内完成
- 部署过程中每一步都有明确结果与失败提示

### Phase 4：分发与安装包

目标：把项目从“源码部署”升级为“可直接分发给普通用户”的版本。

包含内容：

- 免 Python 的打包版

建议原因：

- 这是收益很高的一步，但必须建立在前面流程已经稳定的前提上
- 否则每次打包都只是把问题一起封装出去

交付物：

- Windows 可执行发行版
- 安装后可直接运行初始化和主程序
- 打包时明确处理：
  - Python 运行时
  - 项目依赖
  - `mitmproxy` 依赖或独立运行方案
  - 本地配置文件和日志目录

验收标准：

- 用户机器上不需要手动安装 Python
- 通过安装包即可完成首次部署

## 推荐实施顺序

如果按具体任务编号排序，我建议是：

1. 更清晰的运行状态 UI
2. 环境体检页
3. 一键导出诊断包
4. README 分层文档
5. 自动生成本地配置文件
6. 模板配置产品化
7. 首次运行时自动截取并生成本机模板
8. 错误自动修复
9. 一键初始化脚本
10. 首次启动向导
11. 免 Python 的打包版

## 为什么不是先做打包版

不建议一开始就做打包版，原因是：

- 当前最脆弱的点不在“安装 Python”，而在“新版微信的采集链路是否稳定”
- 如果先打包，问题不会消失，只会更难调试
- 打包版最好基于已经稳定的初始化流程和健康检查页，这样维护压力最低

## 建议的成功指标

建议在项目里追踪这些指标：

- 新机器首次部署成功率
- 首次部署平均耗时
- `cv` 模式首次采集成功率
- 需要人工重采模板的比例
- 需要查看日志才能解决的问题比例
- 使用缓存 `openid` 启动的比例

## 当前状态备注

基于本次实际部署经验，当前最值得优先解决的痛点是：

- 新版微信界面差异导致默认模板不稳定
- 微信会话已选中时重复点击会取消选中
- 浏览器残留窗口会影响下一次采集
- 用户缺少一眼看懂当前状态的页面

所以本计划优先把“可观测性 + 模板与采集稳定性”排在最前面。
