# Implementation Backlog

## 使用方式

这份 backlog 用于把 [PROJECT_PLAN.md](/D:/wei-class/PROJECT_PLAN.md) 落到可以逐项执行的任务上。

执行原则：

- 任务默认由 Codex 负责实施、验证、整理变更
- 用户只在关键决策点参与
- 所有任务尽量以“小步可提交”的方式推进
- 每个任务完成后都应满足对应验收标准，再进入下一个任务

## 协作约定

### Codex 负责

- 阅读代码、设计方案、实施改动
- 运行本地验证
- 整理 diff 摘要
- 提出建议的 commit 边界
- 维护 backlog 状态

### 用户负责

- 在关键产品决策上拍板
- 在适合的时机决定是否 commit
- 在需要人工交互的软件界面上做最少量操作
- 在发布或分发前做最终确认

## 提交节奏

建议每个 Phase 至少拆成 2 到 4 个提交，而不是一个大提交。

推荐的提交粒度：

- 一类能力一个提交
- 一次结构性重构一个提交
- 文档和脚本尽量分开提交
- 能独立回滚的内容尽量单独提交

## 状态标记

- `todo`：尚未开始
- `doing`：正在进行
- `blocked`：被依赖或决策阻塞
- `review`：已完成开发，等待用户确认是否提交
- `done`：已提交或明确完成

## Phase 0：可观测性与支持能力

### P0-01 统一运行状态模型

- 状态：`review`
- 目标：把首页、健康检查、日志摘要、诊断导出需要的状态统一到一套结构中
- 主要工作：
  - 盘点当前 `/api/openid_status`、`/api/check_session`、日志文件、mitm 状态来源
  - 定义统一状态模型
  - 抽出状态聚合层，避免页面和脚本各自拼状态
- 输出：
  - 统一状态对象
  - 统一状态获取函数
- 验收标准：
  - 页面、诊断包、初始化脚本都能复用同一套状态数据
- 依赖：无
- 建议提交名：
  - `feat: unify runtime health state model`

### P0-02 环境体检页与健康检查接口

- 状态：`review`
- 目标：给用户一个可视化的部署检查页
- 主要工作：
  - 新增健康检查接口
  - 检查项至少包含：
    - Python 环境
    - 微信窗口状态
    - 系统代理状态
    - `mitmproxy` 监听状态
    - 证书状态
    - 模板状态
    - `latest_openid.json` 状态
  - 新增前端体检页
- 输出：
  - `/api/health` 或等价接口
  - Web 体检页面
- 验收标准：
  - 用户不看终端也能判断是哪一步没完成
- 依赖：`P0-01`
- 建议提交名：
  - `feat: add deployment health checks and diagnostics page`

### P0-03 首页运行状态增强

- 状态：`review`
- 目标：让首页直接展示关键运行状态
- 主要工作：
  - 展示：
    - `openid` 来源
    - 上次刷新时间
    - 下次刷新时间
    - 最后错误
    - 当前采集方式
    - 当前是否使用缓存回退
    - mitm 状态
  - 对错误提示做更面向用户的文案整理
- 输出：
  - 首页状态区块
- 验收标准：
  - 首页能回答“现在能不能用、为什么不能用”
- 依赖：`P0-01`
- 建议提交名：
  - `feat: improve runtime status visibility on home page`

### P0-04 诊断包导出

- 状态：`review`
- 目标：失败时一键导出排查材料
- 主要工作：
  - 导出脱敏日志
  - 导出运行状态 JSON
  - 导出模板状态和配置快照
  - 设计诊断包目录结构
- 输出：
  - 导出接口或脚本
  - 诊断包格式文档
- 验收标准：
  - 用户能把一份诊断包交给开发者，不需要手工截图和复制日志
- 依赖：`P0-01`
- 建议提交名：
  - `feat: add support bundle export`

### P0-05 文档分层重构

- 状态：`review`
- 目标：降低用户阅读 README 的负担
- 主要工作：
  - 产出：
    - `README.quickstart.md`
    - `README.troubleshooting.md`
  - 主 README 保留全量说明和导航
- 输出：
  - 分层文档结构
- 验收标准：
  - 新用户可以优先走 quickstart，出错时再看 troubleshooting
- 依赖：`P0-02`
- 建议提交名：
  - `docs: split quickstart and troubleshooting guides`

## 每次迭代的标准流程

每次我们推进一个 backlog 任务时，采用固定节奏：

1. 我先阅读相关代码和当前状态
2. 我直接实现任务，并自行完成本地验证
3. 我向你汇报：
   - 改了什么
   - 风险点
   - 建议的 commit 边界
4. 你只需要决定：
   - 现在 commit
   - 继续合并到下一批再 commit
5. 如果你决定 commit，我再给你整理标准的提交说明
